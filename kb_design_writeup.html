Building the Knowledge Base for a Multi-Agent Mining Investigation System

From raw expert reports and regulatory documents to structured, agent-ready knowledge — design decisions, architecture, and expansion strategy.

By Ridwan A. Adebayo  ·  Supervisors: Anna Markarian & Igor Temkin

When you set out to build an AI system that investigates mining accidents, one of the first things you discover is that the intelligence of the system is only as good as the knowledge it can draw on. Before any agent reasons, before any argumentation graph is computed, there is a more fundamental question: what do we actually need to store, in what form, and why?

This write-up documents the knowledge base design for a multi-agent investigation system built as part of an MSc thesis at NUST MISIS. It traces the thinking behind every structural decision — from the choice to decompose expert reports into atomic arguments, to the two-file architecture, to the planned expansion strategy across three source tiers.

§ 01 The Problem with Raw Documents

The starting point was three expert investigation reports on the Kostenko mine explosion (Kazakhstan, October 2023, 46 fatalities). These were DOCX files in Russian — each written by a different expert team with a different analytical lens, each reaching different conclusions on the same incident.

The instinct is to feed these documents directly into the system. Vector-embed them, store them in a retrieval database, let agents query them. This works for general question-answering. But it does not work for what this system needs to do.

The system does not need to find text about the ignition source. It needs to know that Expert A claims the ignition was the angle grinder, Expert B claims it was AFC chain friction, and Expert C says the ignition source cannot be determined — and that these three positions are in direct conflict with each other. Design decision log, March 2025

That kind of structured, relational knowledge — who claims what, supported by what evidence, with what confidence, and in conflict with whom — cannot be reliably extracted at query time from raw text. It needs to be explicit in the data structure itself.

This is the core reason the KB does not store documents. It stores arguments.

§ 02 The Argument as the Fundamental Unit

Every piece of investigative knowledge in the system is represented as a structured argument — a formalization borrowed from Toulmin's model of argumentation. Each argument captures not just a conclusion, but the evidence it rests on and the reasoning that connects evidence to conclusion.

The schema has eight fields:

id Unique identifier, e.g. U-A3, K-A4, D-A9
source Which expert or agent produced this argument
topic The investigation question it addresses, e.g. "Ignition source"
claim The conclusion being asserted
evidence The data or observations supporting the claim
warrant The reasoning connecting evidence to claim
confidence Numeric confidence score (0.0–1.0)
cause_categories Taxonomy labels for classification and matching

A real example, extracted from the Kolikov-Meshcheryakov expert report:

{ "id": "K-A4", "source": "K", "topic": "Ignition source", "claim": "Most probable ignition source was sparking from the armored face conveyor (AFC) chain operating in the methane-rich zone.", "evidence": "Witness testimony from 4 protocols confirming sparking during conveyor operation; physical tests confirming conveyor sparking is possible; sparking zone coincides with floor-level methane measurements of 1.2–17.9%.", "warrant": "Documented as major ignition cause in 16% of Russian coal mine explosions 2004–2019 (N=50 events, 645 casualties). Conveyor sparking zone overlaps methane accumulation zone.", "confidence": 0.72, "cause_categories": ["ignition_source_mechanical", "methane_accumulation"] }

This argument is directly comparable with U-A3 (Usembekov's angle grinder hypothesis) and D-A5 (DMT's "ignition source unknown" conclusion). The system can detect that conflict automatically because all three share the same topic field — and the explicit claims can be fed to the argumentation layer for formal resolution.

This would not be possible if the knowledge were stored as raw text.

§ 03 The Pipeline: From Documents to Arguments

The journey from raw expert reports to agent-ready structured arguments happens in the first subsystem of the architecture — v1 (Decomposition). Understanding this pipeline is essential to understanding why the KB looks the way it does.

Input
Raw documents (DOCX, PDF, sensor logs)
v1 — Mode 2
LLM-assisted argument extraction
v1 — Mode 1
Structured JSON input (pre-extracted)
Output
Uniform 8-field argument schema
Downstream
v2–v6 agents, all mode-agnostic

v1 operates in two modes. In Mode 1 — the primary mode used for both test cases — the arguments have already been manually extracted and structured into JSON. The system loads and validates them directly. In Mode 2, a raw text passage from an investigation report is fed to an LLM, which extracts structured arguments and maps them to the 8-field schema.

Key design property
Both modes produce identical output schema. Everything downstream — classification, precedent matching, agent reasoning, argumentation — is completely mode-agnostic. The KB does not know or care whether its arguments came from manual extraction or LLM processing.

The "heterogeneous data sources" language in the thesis proposal refers to this pipeline capability — that v1 can, in principle, accept sensor logs, maintenance records, or witness transcripts and convert them into the same uniform argument format. For the prototype, this is demonstrated through Mode 2 evaluated against Mode 1 as ground truth.

§ 04 Two Files, Two Roles

The knowledge base consists of two JSON files. They serve fundamentally different purposes in the system, and conflating them is a common source of confusion.

File Role Used by Content
kostenko_knowledge_base.json Case-specific evidence — the input to the investigation v4 specialist agents (reasoning over) 17 arguments from 3 expert reports; 4 attack relations; 5 support relations
rostechnadzor_regulatory_kb.json Background knowledge — the context for reasoning v3 (precedent matching), v4 Agent 4 (regulations), v1/v2 (taxonomy) 7 precedent cases; regulatory requirements; cause taxonomy; violation patterns

The Kostenko file is not a precedent. It is the case under investigation. The 17 arguments it contains are what the four specialist agents in v4 will reason over, attack, and support. The argumentation graph is built from these arguments.

The regulatory KB, by contrast, is what agents consult when they need context — what similar accidents looked like, what regulations were in force, what the standard causes are. It is the system's background knowledge about the domain, not about the specific incident.

The test case is the question. The regulatory KB is the domain knowledge needed to answer it.

§ 05 What Is Currently in the Regulatory KB — and What Is Missing

The regulatory KB currently contains seven precedent cases, all sourced from Russian Rostechnadzor annual reports (2020–2024). It also contains regulatory requirement summaries, a two-tier cause taxonomy (10 technical + 7 organizational categories), and recurring violation patterns.

Seven cases is workable for a proof of concept but thin for a thesis that claims to demonstrate meaningful precedent matching. The quality of v3's output scales directly with the breadth and relevance of cases in the precedent store.

There is also a structural gap: Rostechnadzor covers only Russian Federation mines. The Kostenko mine is in Kazakhstan. While it operates under an equivalent regulatory framework and has identical methane hazard classification, several directly comparable incidents — including Kostenko itself — fall outside Rostechnadzor's jurisdiction entirely. A system that cannot surface Listviazhnaya 2021 as a precedent when investigating Kostenko 2023 is missing its most important comparator.

Confirmed: Alardinskaya 2024 is the closest precedent currently in the KB
Roof fall in the goaf displacing ignited methane-air mixture into longwall workings — almost identical mechanism to Kostenko. But Listviazhnaya 2021 (51 killed, methane explosion, same regulatory baseline in force during Kostenko) is not currently structured as a searchable precedent.

§ 06 Knowledge Base Expansion — Three-Tier Strategy

The expansion goal is a minimum of 20 total precedent cases, prioritizing underground coal mines with methane and explosion incidents, but covering the full accident type spectrum the system is designed to handle.

Tier Source Cases available Language Status
Tier 1 MSHA Major Disaster Investigations (USA) ~5 high-quality cases (UBB 2010, Sago 2006, Jim Walter 2001, Crandall Canyon 2007, Quecreek 2002) English To be extracted
Tier 2 Rostechnadzor annual reports (already loaded) 7 cases structured Russian Partial — expand within reports
Tier 3 International incidents via academic literature (Turkey, Poland, China, Ukraine, New Zealand) ~10+ cases documented in English papers Mixed (English academic) To be identified and extracted

Tier 1 — MSHA

MSHA's Major Disaster Investigations are the highest-quality English-language source available. For each listed disaster, MSHA publishes a full investigation report, an internal review, and in several cases an independent NIOSH assessment. The Upper Big Branch explosion (2010, 29 fatalities) is already planned as the system's secondary test case — but several other MSHA disasters are strong candidates for the precedent KB: Sago Mine (2006, methane ignition triggering explosion), Jim Walter Resources (2001, two methane explosions within one hour), and Crandall Canyon (2007, coal outburst with seismic contribution).

Tier 3 — International incidents

For accidents outside MSHA and Rostechnadzor jurisdiction, the strategy is to source structured information from academic papers rather than official investigation databases. The Soma mine disaster (Turkey, 2014, 301 fatalities — spontaneous combustion and carbon monoxide) and Pike River (New Zealand, 2010, 29 fatalities — methane explosion) are both extensively documented in English-language mining safety literature. While the source-level detail is lower than MSHA reports, these cases provide geographic diversity and cover accident types and failure modes underrepresented in the Russian and American data.

Schema note
All new precedents will follow the existing schema: technical_causes, organizational_causes, violated_regulations, cause_categories, relevance_to_kostenko. The regulatory reference field will note the source framework (30 CFR for US cases, national equivalents for others) rather than forcing all cases into Russian regulatory terminology. This preserves source fidelity while maintaining structural uniformity.

§ 07 What the KB Deliberately Does Not Contain

One category is conspicuously absent from both KB files: a dedicated domain knowledge store — mining engineering textbooks, equipment manuals, gas physics references. This is a deliberate design decision, not an oversight.

The system's specialist agents access domain knowledge through the LLM's training data. An LLM like GPT-4 or Claude has substantial embedded knowledge about methane explosion dynamics, ventilation engineering, and coal seam geology. Adding a full mining engineering knowledge base to the KB would be an enormous extraction effort with marginal benefit for a prototype that already has a capable LLM doing the reasoning.

This is acknowledged as a limitation in the thesis. A production system — one deployed by a mining regulatory body or operator — would benefit significantly from a verified, up-to-date domain knowledge store. For the research prototype, the LLM's embedded knowledge is sufficient to demonstrate the architecture's value, and the limitation is stated explicitly rather than papered over.

§ 08 Where This Sits in the Larger Architecture

The KB is not a subsystem of the investigation pipeline — it is the backbone that all six subsystems draw from. v1 uses the cause taxonomy for labeling. v2 uses accident type definitions for classification. v3 queries the precedent store. v4 agents receive relevant regulations and precedents via RAG. v5 builds the argumentation framework from the case-specific arguments. v6 generates the report from v5's output.

In Markarian's original architecture, the knowledge base is modeled as a central node connected to all six subsystems via bidirectional edges in a directed graph G = {V, E}. The two-file design here is a direct implementation of that pattern — one file per knowledge type, each serving the subsystems that need it.

The expansion of the precedent KB from 7 to 20+ cases is, in this sense, not just a data collection exercise. It is the difference between a system that can demonstrate meaningful precedent retrieval and one that retrieves the same few cases regardless of the accident under investigation. The argumentation layer is only as interesting as the diversity of the cases it can draw on for context.
